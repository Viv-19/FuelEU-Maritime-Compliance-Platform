# AI Agent Workflow Log

This document describes how AI agents were used during the development of the **FuelEU Maritime Compliance Platform**.

The system was developed using a **phase-based workflow**, where each module of the application was implemented incrementally using AI-assisted development.

The development approach combined:

* domain research
* architecture planning
* prompt engineering
* AI-assisted code generation
* manual validation and debugging

---

# Agents Used

| Agent                             | Role                                                                              |
| --------------------------------- | --------------------------------------------------------------------------------- |
| **ChatGPT**                       | Domain research, architecture planning, prompt engineering                        |
| **ChatGPT (Deep Research)**       | Generated a phase-wise development plan                                           |
| **Antigravity (Google DeepMind)** | Primary coding agent used for scaffolding, implementation, testing, and debugging |

### How the Agents Were Combined

The workflow followed a **multi-agent collaboration pattern**:

1. ChatGPT was used to understand the **FuelEU Maritime domain and system requirements**.
2. ChatGPT Deep Research generated a **phase-wise engineering plan**.
3. ChatGPT then generated **structured prompts for Antigravity**.
4. Antigravity implemented the code according to these prompts.
5. The generated code was **reviewed, validated, and corrected manually**.

This approach allowed structured development while maintaining architectural consistency.

# Example 1 — Domain Modeling (Phase 1)

### Prompt

```
You are a Staff-Level Backend Engineer working on a production-quality system.

Your task is to implement the DOMAIN LAYER for a FuelEU Maritime Compliance Platform.

This phase focuses ONLY on domain modeling.

Do NOT implement HTTP APIs, database logic, or infrastructure.

The goal is to create a clean and framework-independent domain layer.
```

The prompt then defined the required domain entities:

* Route
* ShipCompliance
* BankEntry
* Pool
* PoolMember

and included strict architecture rules such as:

* Domain layer must not import Express, PostgreSQL, or React
* Only TypeScript and language features may be used
* Entities must include validation logic

### Generated Output (Excerpt)

```typescript
export class Route {
  constructor(
    public readonly routeId: string,
    public readonly vesselType: string,
    public readonly fuelType: string,
    public readonly year: number,
    public readonly ghgIntensity: number,
    public readonly fuelConsumption: number,
    public readonly distance: number,
    public readonly totalEmissions: number,
    public readonly isBaseline: boolean
  ) {
    if (!routeId) throw new Error("routeId cannot be empty")
    if (year < 2000) throw new Error("Invalid year")
    if (ghgIntensity <= 0) throw new Error("ghgIntensity must be positive")
  }
}
```

### Usage

The generated entities were used as the **core domain models** of the backend application.

These models represent real FuelEU concepts such as:

* route emission records
* compliance balances
* banking entries
* pooling members

Unit tests were also generated to validate entity behavior.

The prompt specified tables including:

* bank_entries

* pool_members

and required migration files and seed data.

### Generated Output (Excerpt)

Example migration:

```sql
CREATE TABLE routes (
 id SERIAL PRIMARY KEY,
 route_id VARCHAR(20) UNIQUE NOT NULL,
 vessel_type VARCHAR(100) NOT NULL,
 fuel_type VARCHAR(50) NOT NULL,
 year INTEGER NOT NULL,
 ghg_intensity NUMERIC(10,4) NOT NULL,

```

### Usage

The generated migrations were used to create the database schema.

Seed scripts inserted the dataset provided in the assignment specification.

After running:

```
npm run db:migrate
npm run db:seed
```

the system successfully populated the `routes` table with the provided dataset.

# Example 5 — Integration Debugging

### Prompt

```
The frontend shows "Failed to fetch".

Perform a full integration debug and fix the issue between
frontend (port 5173) and backend (port 3001).
```

### Root Cause Identified

The frontend API client was using an incorrect base URL.

The backend also lacked proper CORS configuration.

### Fix

Backend:

```ts
app.use(cors({
 origin: "http://localhost:5173"
}))
```

Frontend API client:

```ts
const API_BASE = import.meta.env.VITE_API_URL || "http://localhost:3001"
```

After updating `.env`:

```
VITE_API_URL=http://localhost:3001
```

the frontend successfully fetched route data.


# Best Practices Followed

### Phase-Based Development

Instead of generating the entire system in a single prompt, development was divided into phases:

1. Project scaffolding
2. Domain modeling
3. Database layer
4. API implementation
5. Frontend scaffolding
6. Feature modules
7. Integration and debugging

This reduced hallucinations and improved consistency.

---

### Architecture Constraints in Prompts

Each prompt explicitly enforced:

* hexagonal architecture
* strict TypeScript typing
* framework independence for domain logic

This ensured architectural correctness.

---

### AI for Structure, Human for Validation

AI agents generated structure and boilerplate code.

Human review ensured:

* regulatory compliance
* architectural correctness
* functional integration
